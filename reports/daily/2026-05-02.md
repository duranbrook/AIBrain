### Today's actual top 3 learnings (corrected — supersedes the single meta-learning above)

1. **iOS SourceKit's cross-file diagnostics are systemically unreliable in this project; ground-truth is `xcodebuild`, not the IDE.** SourceKit consistently flagged "Cannot find type X in scope" for types defined in sibling files (`APIClient`, `ChatHistoryItem`, `Agent`, `UIImagePickerController`, `UIColor`) during the iOS technician chat build, even after the project compiled cleanly. The mechanism: SourceKit indexes files in isolation without full project context; Xcode's build system resolves cross-file references correctly. Practical implication: when working in an iOS Swift project (especially one with many sibling files referencing each other across `Network/`, `Models/`, `Views/`), do not treat SourceKit warnings as build failures. Run `xcodebuild ... build 2>&1 | grep "error:\|BUILD"` and gate "this is broken" claims on that output, not on IDE squigglies. This pattern repeats across other Apple-platform projects too (Mac/iOS/visionOS Xcode workspaces with module boundaries). Saved as memory: `feedback_ios_sourcekit.md`.

2. **Alembic migration heads can be hash-named files, not numbered files — always run `alembic heads` (or read the latest migration's `revision` field) before authoring a new migration.** When migration 0024 was authored for the `vehicle_id` FK on `quotes`, it set `down_revision="9c5e490936db"` (the hash-named file) instead of `"0023"`. This silently forked the chain — `alembic upgrade head` would have failed in the same multi-head way schema migrations classically do. Caught only in code review, not by tests. Two takeaways: (a) the alembic chain in this project mixes numbered (`0001`–`0023`) and hash-named (`9c5e490936db`) revision IDs, with the latter sitting *between* numbered files in the middle of the chain — a subtle trap for any migration author scanning the directory by filename; (b) `alembic heads` is the canonical "what is the actual tip?" answer and should be a hard pre-step in any "create migration" workflow. Saved as memory: `alembic_migration_chain.md`.

3. **Owner stated the explicit product vision — three pillars: AI Technician (built), Owner AI Crew (specialist crew of agents, not a single bot), Consumer Vehicle History + Connected Ecosystem (two-sided).** This is the most strategically significant input of the week and reframes how to design owner-facing features for the next phase. The "AI Crew" framing is not a single multi-agent assistant but *multiple* specialist team members the owner converses with by domain (parts, revenue, scheduling, etc.) — the design metaphor is "who on your team handles this?" not "which module do I open?". The consumer-facing surface is a separate product surface, not a view inside the shop tool: car owners can query their own vehicle's service history, and the connected ecosystem makes any car's history queryable from both sides. The connected-ecosystem sentence was cut off mid-thought and needs follow-up to fully scope.
