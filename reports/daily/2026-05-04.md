## Today's Top 3 Learnings (2026-05-04)

1. **The S3 presigned-URL trifecta — three independent failure modes that all hit in one debugging session.** Photo-to-finding pipeline shipped today required all three fixes to land before the end-to-end flow worked: **(a) Never store presigned S3 URLs as DB identifiers** — presigned URLs include a time-limited signature that rotates every call, so any string comparison between a stored presigned URL and a freshly-generated one fails. The bug surfaced as: photos uploaded fine, but the PDF rendered all of them in the "unassigned gallery" instead of attached to the right finding, because `assigned_photo_urls` (computed from stored presigned URLs in `findings.photo_url`) never matched `media_urls` (freshly presigned at render time). Fix: always store the *stable* S3 URL (`https://pitagents.s3.us-east-1.amazonaws.com/session/media/file.jpg`) in DB rows; presign at render/fetch time only. **(b) S3 buckets created post-2020 reject SigV2 with HTTP 403** — `boto3.client("s3").generate_presigned_url(...)` defaults to legacy Signature V2 (`AWSAccessKeyId=...` query format). Fix: pass `Config(signature_version="s3v4")` when constructing the client. **(c) SigV4 has a hard 7-day max expiry (604,800 seconds)** — passing `expires_in=2_592_000` (30 days) returns 400 `AuthorizationQueryParametersError`. The trap is that SigV2 *did* allow longer expiries, so any code path that previously worked with SigV2 + 30-day expiry now silently breaks twice when migrated to SigV4. Fix: cap to 7 days (or fewer) for any presigned URL serving externally. All three are now persisted in `sync/memory/-Users-joehe-workspace-projects-pitagents/feedback_presigned_url_pitfalls.md`. **Why this is a real today-learning:** the bug had two distinct manifestations (Anthropic 400 "Unable to download" when chat sent images; iOS AsyncImage 403 when displaying findings), both with the same root cluster, both surfacing only at runtime in prod against a real S3 bucket. The shared lesson — *presigned URLs are renderings, not identities* — is the load-bearing insight; the SigV4 + 7-day caps are corollaries that fall out of using S3 in 2026.

2. **`asyncpg` misparses `::jsonb` cast immediately after a named parameter — use `CAST(:p AS jsonb)` instead.** Migration 0029 had `tools = :tools::jsonb` which caused **3 consecutive Railway deploy failures** before Joe diagnosed it. The asyncpg server-side prepare protocol parses `:tools` as a named param and then sees `::jsonb` adjacent — but the Postgres syntax `expr::type` for casting and the leading `:` of named params look identical to the lexer at that boundary, so the named-param substitution captures the colons. Fix: rewrite as the explicit functional form `tools = CAST(:tools AS jsonb)`, which has no boundary ambiguity. **Why this matters beyond this one deploy:** any alembic migration that uses `::jsonb` (or `::uuid`, `::timestamp`, etc.) immediately after a `:named_param` will break under asyncpg, and the failure mode is "deploy fails on apply with a SQL parse error" — which on Railway means the whole service won't start. There was no automated test that would have caught this; the DB pings only after the migration runs. **The general rule for migration SQL:** prefer `CAST(... AS type)` over `::type` *whenever* the cast operand is a named parameter, because the colons collide with asyncpg's substitution. Worth adding to a project-level conventions note for migrations.
