The morning had only 1 learning (car-parts market research). The 9am execution run generated three substantial technical outputs. Here are the final top 3 learnings for the full day:

#### 1. RAG chunking strategy is the highest-leverage architectural decision in a RAG system
**Category:** Technical

Chunking determines both retrieval quality and context utility. The key insight from the deep dive: fixed-size chunking is simple but semantic-boundary-aware approaches (recursive character splitting, semantic chunking, or structure-aware chunking) significantly improve retrieval relevance. The "parent document retrieval" pattern is particularly powerful — embed small chunks for precise retrieval, but return the full parent document to the LLM for rich context. Chunk overlap (e.g., 64-token overlap on 512-token chunks) preserves cross-boundary meaning that would otherwise be lost. Reference: `docs/learning/rag-deep-dive.md`

#### 2. Two-stage interviews (take-home + live) provide complementary and non-overlapping signal
**Category:** Process / Collaboration

Take-home reveals what a candidate produces with time, autonomy, and no eyes on them — real-world code quality, README communication, architectural judgment. Live interview reveals how they think under pressure, handle ambiguity, and communicate in real time. Neither alone gives the full picture. The scoring threshold design is also important: gating on Correctness + Communication (not just overall average) prevents a candidate who is technically strong but communicates poorly from slipping through. Reference: `docs/interview-design/coding-interview-rubric.md`

#### 3. Platform gap analysis: missing schema models are the hidden critical path, harder to spot than missing UI features
**Category:** Technical / Architecture

In the PartFinder gap analysis, 10 missing schema models were identified that block entire feature areas — StorageLocation for yard tracking, Dispute for dispute lifecycle, SellerTier for reputation, FraudFlag for moderation, etc. These are invisible from the UI but are blockers for anything built on top of them. The lesson: when assessing a platform's completeness, audit the data model first before counting features. Schema gaps are higher-leverage and harder to retrofit than UI gaps. Reference: `docs/car-parts/implementation-phases.md`
