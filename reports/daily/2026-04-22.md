## Key Learnings (Top 2)

_Honestly reporting 2, not 3. The three stlab-specific technical learnings from the morning plan belong to **2026-04-21** (that's when the pipeline work actually happened, even though it surfaced to AIBrain today). Re-counting them here would be double-booking._

1. **Nuke-then-reingest is the correct reset after a pipeline shape-change.** After the morning's wave of stlab ingestion fixes landed (code-fence stripping, NullPool/Celery, recursive parser, `on_conflict_do_nothing` upsert, model alias), the very first action of the next stlab session was `/nuke-test-data` — Postgres truncate + Redis flush + MinIO clear, scripted as a single command. That's the right pattern when idempotency isn't yet proven end-to-end: don't patch partial state, wipe and replay from zero. The generalizable rule: **if a pipeline changed shape, the next run starts from empty, not from stale partials.** Having a one-shot reset command (vs. ad-hoc cleanup) is what makes this cheap enough to do routinely. *Source: `sync/journals/interview_stlab/2026-04-22.md`.*

2. **An 11-day carry-over P1 is a signal to demote, not to push harder.** The 09:00 execute run formally moved car-parts contact enrichment from P1 to P2 after 11 consecutive zero-progress days. The trigger wasn't new information about the task — the task hadn't changed — it was the *arrival of genuine momentum in a different project* (stlab). Backlog hygiene beats backlog sentimentality: **when a task has been P1 for ≥2 weeks with no commits and there's active momentum elsewhere, demote on that signal alone.** Re-scope before promoting again (here: top-10 chains as validation chunk, not the full 3,623). *Source: `tasks/backlog.md`, commits `3825f5f` + `11a09cb`.*
