## Key Learnings (Top 3)

1. **In a hold-to-talk button over WebRTC, browser event order — `mousedown → mouseup → click` — will cancel the model's tool response unless you guard the click handler.** Joe's 2026-04-30 16:00 session diagnosed why the voice control showed "connected" (green ring) but never navigated: the natural sequence in PTT mode is `mousedown` → `startCapture()` (audio track on), `mouseup` → `stopCapture()` (audio committed to OpenAI), `click` → `disconnect()` — and the click fires *before* OpenAI sends back the tool call, killing the session early. Fix: a `mouseDownAt` ref to track hold duration; holds > 200ms suppress the click-handler's `disconnect()`, while short taps still toggle off intentionally. Generalizes to any "hold-to-X, tap-to-Y" affordance over an async backend: don't put the X-completion handler on `click`, and if you must, gate it on a duration measured from `mousedown`. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session 16:00.

2. **OpenAI Realtime tool chaining: `tool_choice: 'auto'` is correct, `'required'` traps you in the depth limit, and `responseId` from `response.output_item.done` is unreliable for chain-state tracking.** Joe's 2026-04-30 final session diagnosed why a multi-step voice command ("go to reports and find the Honda Civic") was firing tool 1 and then stalling: (a) the chain condition `(pendingCalls.length > 0 || executedCount > 0)` evaluated `executedCount=0` because `responseId` from `output_item.done` was sometimes null, so the `#responseToolCounts` map never got populated for that response — so `#requestPostToolResponse` never fired. Fix: add a session-scoped `toolExecutedDuringResponse` boolean, set inside `#executeToolCall`, reset on every `response.created` — a more reliable proxy for "a tool ran in this response" than counting by responseId. (b) `tool_choice: 'required'` was forcing the model to keep calling tools even when the chain was logically complete, burning through `depth=8`. `'auto'` lets the chain terminate naturally. (c) Updated INSTRUCTIONS to "call ALL needed tools in a single response" so the model batches in response 1 and stops on the post-tool `response.create`. Source: `sync/journals/projects_pitagents/2026-04-30.md` final session.

3. **A Next.js Vercel project with `framework: null` silently uses `@vercel/static-build` — only `public/` assets ship, and prod 404s on every dynamic route.** Joe's 2026-04-30 15:20 session: `pitagents.vercel.app` was returning 404 because the Vercel project had been created with `framework: null` instead of `nextjs`. Vercel's default builder for null-framework is `@vercel/static-build`, which deploys only static files (the SVGs in `public/`) — the Next.js bundle never goes up. *Plus* a second axis of failure: the team had been deploying to a different Vercel project (`web`) all along, not `pitagents`. Fix sequence: re-link `web/` → `pitagents` project, set `framework: nextjs` via Vercel API, pull settings, rebuild, deploy prebuilt 2.3MB artifact. Two takeaways for future Vercel-deploy hygiene: (a) always confirm `vercel link` points to the project the URL implies; (b) inspect `framework` in project settings — `null` is silently destructive for SSR apps. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session 15:20.
