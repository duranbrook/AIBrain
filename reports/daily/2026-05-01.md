## Today's Top 3 Learnings (2026-05-01) — finalized by 20:00 evening retro

These are the three most significant *new* learnings from today's actual work (the sleek redesign + dashboard pipelines that shipped to `pitagents.vercel.app` during the morning, and the AIBrain cycle-collision pattern that surfaced overnight). They supersede the bonus list flagged in the midday update; the canonical 04-30 top-3 below is preserved unchanged because this report still doubles as the retro for that day.

1. **Next.js client/server boundary trap: storing React component functions inside a "data" object silently forces every consumer of that object client-side.** In `web/components/dashboard/tiles.tsx`, each `GROUPS[i].tiles[j].icon` field stores a React component function (`MyIconComponent`). The whole `GROUPS` constant *looks* like plain JSON data — and is otherwise serializable — but a component-function reference cannot cross the Next.js server/client boundary. The pragmatic fix Joe landed: keep `tiles.tsx` itself unmarked (so it stays server-renderable as data), and let only the consumers that actually invoke `<icon />` (`FeatureTile`, `GroupSection`) be marked `'use client'`. Generalizes: anything you store in a "config" object that ends up rendered as JSX is a hidden client-boundary tripwire — keep the *invocation* on the client side, not the *registration*. Source: `sync/journals/projects_pitagents/2026-05-01.md` "dashboard implementation".

2. **Double-submit prevention in async login: a `navigated` flag set in the success branch and consumed in the `finally` block is cleaner than disabling the form button.** Sleek-redesign's Login fix: `try { await login(); navigated = true; router.push(...) } finally { if (!navigated) setSubmitting(false) }`. The standard pattern of disabling the submit button immediately after first click flickers when the navigation race resolves and re-enables the disabled state in `finally`. The `navigated`-flag pattern moves the gating into the finally block itself: if we successfully kicked off navigation, we never re-enable. Generalizes to any `try { side-effectful-thing-that-changes-route(); navigated = true } finally { ... }` — one boolean, no extra state, no flicker. Source: `sync/journals/projects_pitagents_.worktrees_feat-sleek-redesign/2026-05-01.md` Task 6.

3. **Concurrent-session collision on `.git/*.lock` is the failure mode when AIBrain's scheduled tasks overlap.** Today the morning-plan and evening-retro fired simultaneously (both were delayed by the cycle-scheduler gap). They raced on `.git/*.lock` files held in read-mode by macOS Xcode's `com.apple.dt.Xcode.sourcecontrol.Git.xpc` (PID 7957). `lsof` showed all four lockfiles held open. Recovery pattern: rename (don't `rm -f`) the locks to `.lock.stale.<ts>` — recoverable forensics, both racing sessions then drain. Sandbox cannot delete files inside the fuse-mounted `.git/` directory, only rename, so `mv` is the *only* clean tool here. Operational consequence: if multiple scheduled tasks are likely to overlap, gate each cycle script behind a `lsof | grep -q '.git/.*\.lock'` pre-check and skip on contention rather than racing. Source: `sync/journals/AIBrain/2026-05-01.md` Session 00:30.

## Carryover Learnings (canonical for 2026-04-30 — retained from morning retro)

1. **In a hold-to-talk button over WebRTC, browser event order — `mousedown → mouseup → click` — will cancel the model's tool response unless you guard the click handler.** Joe's 2026-04-30 16:00 session diagnosed why the voice control showed "connected" (green ring) but never navigated: the natural sequence in PTT mode is `mousedown` → `startCapture()` (audio track on), `mouseup` → `stopCapture()` (audio committed to OpenAI), `click` → `disconnect()` — and the click fires *before* OpenAI sends back the tool call, killing the session early. Fix: a `mouseDownAt` ref to track hold duration; holds > 200ms suppress the click-handler's `disconnect()`, while short taps still toggle off intentionally. Generalizes to any "hold-to-X, tap-to-Y" affordance over an async backend: don't put the X-completion handler on `click`, and if you must, gate it on a duration measured from `mousedown`. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session 16:00.

2. **OpenAI Realtime tool chaining: `tool_choice: 'auto'` is correct, `'required'` traps you in the depth limit, and `responseId` from `response.output_item.done` is unreliable for chain-state tracking.** Joe's 2026-04-30 final session diagnosed why a multi-step voice command ("go to reports and find the Honda Civic") was firing tool 1 and then stalling: (a) the chain condition `(pendingCalls.length > 0 || executedCount > 0)` evaluated `executedCount=0` because `responseId` from `output_item.done` was sometimes null, so the `#responseToolCounts` map never got populated for that response — so `#requestPostToolResponse` never fired. Fix: add a session-scoped `toolExecutedDuringResponse` boolean, set inside `#executeToolCall`, reset on every `response.created` — a more reliable proxy for "a tool ran in this response" than counting by responseId. (b) `tool_choice: 'required'` was forcing the model to keep calling tools even when the chain was logically complete, burning through `depth=8`. `'auto'` lets the chain terminate naturally. (c) Updated INSTRUCTIONS to "call ALL needed tools in a single response" so the model batches in response 1 and stops on the post-tool `response.create`. Source: `sync/journals/projects_pitagents/2026-04-30.md` final session.

3. **A Next.js Vercel project with `framework: null` silently uses `@vercel/static-build` — only `public/` assets ship, and prod 404s on every dynamic route.** Joe's 2026-04-30 15:20 session: `pitagents.vercel.app` was returning 404 because the Vercel project had been created with `framework: null` instead of `nextjs`. Vercel's default builder for null-framework is `@vercel/static-build`, which deploys only static files (the SVGs in `public/`) — the Next.js bundle never goes up. *Plus* a second axis of failure: the team had been deploying to a different Vercel project (`web`) all along, not `pitagents`. Fix sequence: re-link `web/` → `pitagents` project, set `framework: nextjs` via Vercel API, pull settings, rebuild, deploy prebuilt 2.3MB artifact. Two takeaways for future Vercel-deploy hygiene: (a) always confirm `vercel link` points to the project the URL implies; (b) inspect `framework` in project settings — `null` is silently destructive for SSR apps. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session 15:20.

### Supplementary Top 3 Learnings (2026-05-01 late-evening + overnight)

1. **Vercel `vercel.json` placed inside a subdirectory breaks Next.js when Vercel's root is configured to that same subdirectory — even with only `ignoreCommand` set.** When Vercel's root is `web/` (set via dashboard), adding `web/vercel.json` causes every subsequent build to fail with `Couldn't find any pages or app directory`. The harmless-looking `ignoreCommand`-only config silently breaks Next.js's `app/` directory detection. Fix: delete `web/vercel.json` entirely and configure `ignoreCommand` via the Vercel dashboard UI instead (commit `97f0045`). When auto-deploys break (0ms build + Error state), bypass with `cd web && vercel --prod --force`. Generalizes: when a platform is configured to descend into a subdirectory, do **not** add platform-config files inside that subdirectory — config + descended-into-directory is a double-application that breaks framework detection. Source: `sync/journals/projects_pitagents/2026-05-01.md` "seed execution + deployment fixes" + `sync/memory/-Users-joehe-workspace-projects-pitagents/feedback_vercel_nextjs_subdir.md`.

2. **React #185 ("Maximum update depth exceeded") is what's behind a "this page couldn't load" with all-200s — read the browser Console FIRST, not the Network tab.** Joe spent ~2 hours chasing what looked like a deployment race condition (RSC payload / chunk mismatches across two simultaneous Vercel builds) before checking the Console and finding `Minified React error #185`. HTTP status codes only reflect file delivery, not code execution — a JS runtime error caught by the React error boundary renders the same fallback page as a network failure. The actual root cause was almost always traceable to **unstable references in `useMemo`/`useCallback` dependencies**: inline object literals `{ ... }` or `= []` defaults destructured from `data` evaluate to a new reference every render and trigger an infinite `setState` loop. Fix: stabilize the reference (`useMemo`, `useRef`, or a module-level constant). Operational rule for next time: when a deployed page goes blank with all-200s, ask for the browser Console output before assuming a network/deploy issue. Incognito is also a useful disambiguator — eliminates cache as a variable and exposes 401/loading states that may hide the real bug. Source: `sync/memory/-Users-joehe-workspace-projects-pitagents/feedback_react_infinite_loop_debugging.md` (newly added 2026-05-01).

3. **asyncpg rejects non-hex characters in UUID literals — mnemonic test-fixture UUIDs that mix in `v`/`j`/`i`/`p`/`s`/`r`/`q`/`t`/`x`/`m`/`o` will throw at insert time, not at parse time.** Joe's `seed_demo.py` (1,096 lines, 24 tables, 7-customer narrative) used mnemonic 8-char prefixes per table for readability (`vvvvvvvv-...` for vehicles, `jjjjjjjj-...` for job-cards, etc.). The 14-table mnemonic scheme failed on Railway with `invalid input for query argument: invalid UUID` because asyncpg's UUID validator strictly requires `[0-9a-f]`. Commit `1cea1e7` reassigned 13 different prefixes to valid hex equivalents (`b7b7b7b7`, `b8b8b8b8`, `b9b9b9b9`, `c4c4c4c4`, `c5c5c5c5`, `c6c6c6c6`, `c7c7c7c7`, `c8c8c8c8`, `c9c9c9c9`, `d4d4d4d4`, `d5d5d5d5`, `d6d6d6d6`, `d7d7d7d7`). Generalizes for any future seed script that wants mnemonic UUIDs: (a) restrict the alphabet to `[a-f0-9]` from the start (`b7b7...` reads as "vehicle bucket 7" once you internalize it), or (b) use deterministic `uuid5(NAMESPACE, "vehicle-1")` hashes which are valid by construction and human-traceable via the input string. Source: `sync/journals/projects_pitagents/2026-05-01.md` "seed execution + deployment fixes" Session.
