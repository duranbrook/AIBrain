# Daily Report: 2026-04-07

## Top 3 Learnings

### 1. Hybrid retrieval (dense + sparse) via Reciprocal Rank Fusion outperforms either alone
**Source:** RAG implementation spec for PartFinder (`docs/car-parts/rag-implementation.md`)
**Insight:** For domain-specific search (auto parts), pure semantic search misses exact part numbers and OEM codes, while pure keyword search misses natural language queries. The RRF formula `1/(k+rank_dense) + 1/(k+rank_sparse)` with k=60 merges both signals without needing learned weights. This is implementable entirely in SQL with pgvector + `ts_rank` — no external fusion service needed.

### 2. Model tiering by use-case risk profile is the key RAG cost lever
**Source:** RAG spec — choosing Haiku vs Sonnet per use case
**Insight:** Not all RAG use cases need the same model. High-volume/low-stakes queries (parts search, support FAQ) can use Haiku at ~10x lower cost, while high-stakes reasoning (dispute resolution with legal implications) warrants Sonnet. The decision axis is risk-of-wrong-answer, not query complexity. This applies broadly to any multi-use-case RAG system.

### 3. pgvector HNSW with metadata pre-filtering avoids the "retrieve then filter" trap
**Source:** RAG spec — Use Case 1 (parts compatibility search)
**Insight:** A naive approach embeds everything, retrieves top-K by cosine similarity, then post-filters by metadata (year/make/model). This returns fewer than K relevant results when the metadata filter is selective. Pre-filtering in the WHERE clause before the ANN search (supported by pgvector HNSW) ensures all K results match the metadata constraint. Trade-off: the index scan is slightly less efficient, but precision is dramatically better for structured-domain data.
