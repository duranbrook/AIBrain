# Daily Report: 2026-04-28

## Key Learnings (Top 3)

1. **Gherkin feature files written before testing turn ad-hoc smoke into structured spec validation, and surface bugs ad-hoc clicking misses.** Joe's 09:30 Monday session wrote 7 Gherkin feature files in `pitagents/docs/test-plans/` *before* running the smoke on a Pixel 10a. That single run surfaced three distinct bugs that wouldn't have come up in unscripted clicking: (a) `POST /vehicles/{id}/messages` returning HTTP 500 in prod whenever Twilio was flaky, (b) Android `InspectScreen` diverged from spec (showed a single "Start Inspection" button instead of a customer grid), (c) iOS Simulator tap injection driver not reproducing across builds. Generalizes: when smoke-testing manually, articulating each scenario as Gherkin first forces you to enumerate the gaps before clicking — turns "did anything obvious break?" into "did every defined behavior pass?". Source: `sync/journals/projects_pitagents/2026-04-27.md` Session 09:30.

2. **API endpoints must isolate flaky external integrations from durable persistence — optimistic UI in the client cannot paper over backend exceptions.** The Pixel 10a smoke test caught `POST /vehicles/{id}/messages` returning HTTP 500 because Twilio exceptions were propagating uncaught — the message was never saved to the DB at all. Joe's `MessagesViewModel` had clean optimistic-send code (placeholder → replace on success → rollback on failure), but optimistic UI can only smooth over latency, never destroyed durability. The fix (commit `6307b71`): wrap the Twilio/SendGrid call in `try/except`, log a warning, and persist the message with `external_id=null`. Pattern: when a user-facing action *must* succeed but it calls a *best-effort* external service, the API endpoint must isolate the integration in a try/except and persist regardless. Send-and-forget belongs at the API boundary, not in the client. Source: `sync/journals/projects_pitagents/2026-04-27.md` Session 09:30 + 10:10.

3. **iOS Simulator click automation via cliclick + pixel-color calibration is not reproducible across builds.** On 2026-04-26 Joe derived an empirical formula `macOS_y = 79.4 + iOS_point_y × 1.078` for portrait 1080×1920 sim, plus blue-pixel color analysis to find the exact button positions. It worked. On 2026-04-27 the same approach stopped registering as iOS taps — cliclick moved the cursor, but the simulator didn't see them as touches. Keyboard events via AppleScript still worked. Lesson for autonomous mobile testing: pixel-driven taps are fragile across rebuilds (sim instance differences, layout shifts beyond color tolerance). Prefer `xcrun simctl io <UDID> input` + accessibility identifiers, or AppleScript keyboard events where possible. If pixel taps are the only option, plan to recalibrate every build. Source: `sync/journals/projects_pitagents/2026-04-27.md` Session 09:30.
