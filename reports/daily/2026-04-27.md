# Daily Report: 2026-04-27

## Key Learnings (Top 3)

1. **Presigned S3 URLs must never be used as identifiers — store stable S3 URLs in DB, generate presigned fresh at render time.** Storing presigned URLs in `findings.photo_url` caused string comparison between `assigned_photo_urls` (old presigned) and `media_urls` (fresh presigned) to always fail, so every photo fell into the "unassigned gallery" of the inspection PDF. The signature in a presigned URL changes every call, so it can never be a primary key. Generalizes to any S3-backed system: presigned URLs are *capabilities*, never *identifiers*. Captured in `sync/memory/-Users-joehe-workspace-projects-pitagents/feedback_presigned_url_pitfalls.md`.

2. **Verify which API path the client actually uses before fixing the backend.** iOS uses `PUT /quotes/{id}/finalize` for inspection reports — not `POST /sessions/{id}/generate-report` (web-only). Joe spent time fixing `sessions.py` while iOS bugs persisted because the iOS app never hit that code path. The fastest debug check for a "fix landed but bug stays" report is: grep the *client* code for the actual endpoint, then start there. Captured in `sync/memory/-Users-joehe-workspace-projects-pitagents/feedback_ios_api_flow.md`.

3. **Chain-of-thought prompt structure for grouping problems: count first, then map, then output JSON.** For photo-to-finding matching, the working prompt forces Claude to (1) enumerate distinct issues, (2) map photos to enumerated issues, (3) emit JSON — in that order. Skipping the count step caused photos to be over-grouped under fewer findings. Generalizes: when you want Claude to partition things, make it enumerate the partition keys *before* assignment.
