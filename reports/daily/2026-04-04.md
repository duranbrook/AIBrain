## Today's Top 3 Learnings

### 1. Postgres STORED generated columns require immutable expressions — compute time-deltas in application code
**Category:** Technical

Postgres `STORED` generated columns only accept immutable expressions — functions whose output is fully determined by input values alone. `CURRENT_DATE` and similar time-dependent functions are classified as `VOLATILE` (non-immutable), so attempting to use them in a STORED generated column (e.g., `days_until_expiry GENERATED ALWAYS AS (expiry_date - CURRENT_DATE) STORED`) will fail at migration time with a cryptic error. The fix: define the column as a plain nullable integer and compute `days_until` in application code (JS/TS) during data transformation. This is actually cleaner — the derived value only needs to be accurate at read time, not stored.

**Why it matters:** This is a non-obvious Postgres constraint that surfaces only at migration time, not during schema design. Recognizing it early prevents wasted migration slots.

### 2. "Magnet file" conflict pattern in parallel worktree development — plan explicit merge passes between batches
**Category:** Process / Architecture

When running many background agents in parallel via git worktrees, certain files become "magnets" — nearly every agent touches them (e.g., `extractionSummary.tsx` in the zhengyi-he-onsite app). This creates an N-way merge problem: 4 batches × 3 agents = up to 12 concurrent edits to one file. The pattern that works: run agents in bounded batches (3 agents/batch), do a full conflict-resolution pass after each batch merges to main, then launch the next batch. Trying to merge all worktrees at once compounds conflicts beyond what's recoverable with standard 3-way merge tooling.

**Why it matters:** Parallel worktree development is highly efficient for additive features but degrades badly without explicit coordination around shared files. Identifying magnet files upfront (usually the main layout, root component, or primary data transformation file) and planning sequential merge points is essential.

---

### 3. Architectural complexity heuristic for agent delegation: additive vs cross-cutting
**Category:** Process / Collaboration

When deciding whether to delegate a feature to a background agent or implement it in-session, the key heuristic is additive vs cross-cutting. Additive features (new column, new UI card, new endpoint that doesn't change existing ones) are safe to delegate — agents can work independently without needing to know what else is changing. Cross-cutting features that require new DB tables + new API endpoints + new frontend components + integration into existing shared components (like BL-003: lease package amendments with reconciliation API, PackageReconciliation component, and a new Packages tab) should be implemented in-session to avoid coordination overhead, merge conflicts, and architectural drift. The rule of thumb: if a feature touches ≥3 distinct system layers and modifies existing shared components, keep it in-session.

**Why it matters:** Over-delegating cross-cutting features leads to conflicting implementations and requires more time to untangle than doing it directly. Under-delegating additive features wastes in-session time on parallelizable work.
