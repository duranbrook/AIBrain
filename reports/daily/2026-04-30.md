# Daily Report: 2026-04-30

## Key Learnings (Top 3)

1. **Vercel project-level settings are a load-bearing, invisible-from-the-repo failure mode — `framework: null` silently downgrades a Next.js deploy to `@vercel/static-build`.** Production was returning 404s despite local builds passing and the SPA loading fine in dev. Two compounding misconfigs were found in one debug session: (a) the `pitagents` Vercel project had `framework: null`, so `@vercel/static-build` was used instead of `@vercel/next` — only static SVGs from `public/` got deployed, not the Next.js bundle, and (b) prior deploys had been hitting the wrong Vercel project (`web`, not `pitagents`). Fix: re-link `web/` to the `pitagents` project, set framework via Vercel API, pull updated settings, rebuild with `@vercel/next`, deploy the 2.3MB prebuilt artifact. Lesson: `vercel.json` is in the repo, but project-name and framework lock-in are server-side state — when prod 404s but the bundle looks healthy locally, *the first thing to check is the project's framework field on Vercel's side*, not the build output. This is the same class of trap as DNS or env-var drift: invisible from the source tree. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session 15:20.

2. **PTT (push-to-talk) buttons require hold-duration discrimination because the browser always synthesizes `click` after `mouseup`.** Voice control showed a connected green ring but never navigated. The browser fires `mousedown → mouseup → click` for every button press — and the codebase had separate handlers: `mousedown → startCapture()`, `mouseup → stopCapture()` (commits audio buffer to OpenAI), `click → disconnect()`. The `click` *always* fired after `mouseup`, killing the WebRTC session before OpenAI's tool-call response arrived → no network activity, no console output, no tool call. Fix: track `mouseDownAt` ref; if hold > 200ms (PTT release), suppress disconnect inside `click`; if < 200ms (intentional tap), let disconnect run. General rule: any button with both press-hold and tap semantics must measure duration in the click handler, because the click handler has no knowledge of whether it followed a hold or a tap — and browsers don't fire click *instead of* mouseup, they fire it *in addition to* mouseup. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session 16:00.

3. **Multi-step LLM tool chains stall when "did a tool execute this turn" is keyed on an optional streaming-event field — keep an explicit boolean flag set at action time.** Voice multi-step commands (e.g. "go to reports and find the Honda Civic") executed tool 1 then froze. `#handleResponseDone` computed `executedCount = 0` because the `responseId` arriving on `response.output_item.done` could be null, so the per-response counter map `#responseToolCounts` was never populated. The chain condition `(pendingCalls.length > 0 || executedCount > 0)` returned false → `#requestPostToolResponse` never fired → the model never got the chance to emit the next tool call. Three-part fix: (a) add `toolExecutedDuringResponse` boolean, set inside `#executeToolCall`, reset on `response.created` — independent of any responseId; (b) change `tool_choice` from `"required"` to `"auto"` — `required` forced the model to call tools even when the chain was complete, burning the depth=8 limit; (c) instructions updated to "call ALL needed tools in a single response" so the model batches. Pattern: when streaming events can each have null/missing correlation IDs, don't drive state machines off those IDs — keep a flag that's set by the action itself and cleared at known boundaries. Source: `sync/journals/projects_pitagents/2026-04-30.md` Session resumed — Fix voice multi-step chain.
