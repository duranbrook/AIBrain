## Today's Top 3 Learnings

### 1. Anthropic Files API upload requires `MultipartField<InputStream>`, not raw `byte[]`
**Category:** Technical

When uploading files to the Anthropic Files API via the Java/Kotlin SDK (v2.17.0), raw `byte[]` uploads are silently rejected with "Invalid file upload". The correct approach is to wrap the content as a `MultipartField<InputStream>` with an explicit `filename` (e.g., `"document.pdf"`) and `contentType` (e.g., `"application/pdf"`). Separately, `FileUploadParams` in SDK v2.17.0 does not expose a `contentType` metadata field at all — this was confirmed by decompiling the SDK jar with `javap`. For `registerFromBytes` (raw upload endpoint), content type should be derived from context, not passed as a parameter.

**Why it matters:** This is a footgun that wastes hours. The error message doesn't point to the multipart vs byte[] distinction. Knowing the SDK limitation upfront saves debugging time.

---

### 2. Flyway migration management when schema history table is absent
**Category:** Technical / Process

When a Spring Boot app has never run Flyway (no `flyway_schema_history` table exists), the server will fail to start with a migration error even if migrations are logically correct. The pattern to resolve: start with `--spring.flyway.enabled=false`, then apply the migration manually via `docker exec` into the Postgres container. When working across feature branches/worktrees, migration version collisions (e.g., two branches both claiming V60) must be resolved by renaming the later migration — in this case V60 → V63 to skip the versions added in main during parallel development. Making migrations idempotent (`IF NOT EXISTS`) avoids re-run errors when migrating manually applied state.

**Why it matters:** Worktree-based development with database migrations requires extra coordination. The rename-to-skip pattern is a reliable escape hatch.

### 3. GAN-inspired Planner/Generator/Evaluator harness dramatically outperforms solo Claude on long-running app development
**Category:** Architecture / Process

Per Anthropic's March 2026 engineering blog, separating the generator agent from the evaluator agent is the most impactful architectural decision in multi-agent harnesses. Claude is a poor self-QA agent: it "identifies legitimate issues, then talks itself into deciding they weren't a big deal." An independent evaluator using Playwright MCP (testing like a real user) with hard-threshold grading forces honest evaluation. The sprint contract pattern (generator proposes scope + verification method → evaluator reviews before build → test against contract) aligns both agents before work begins. Cost: ~$200 over 6 hours vs $9 solo, but quality is dramatically better for complex UIs.

**Why it matters:** For any serious product-quality frontend or multi-feature app sprint, the harness investment pays off. Solo Claude is fast and cheap but produces mediocre design quality.
