## Week's Top 3 Learnings

### 1. Separating generator and evaluator agents is the highest-leverage architectural decision in multi-agent harnesses

From both the Anthropic engineering blog reference and the zhengyi-he-onsite parallel sprint experience this week: Claude is unreliable as its own QA agent. It identifies legitimate issues, then rationalizes them away. An independent evaluator using Playwright MCP with hard-threshold grading forces honest assessment. The sprint contract pattern (scope + verification proposed upfront, agreed before building) aligns both agents before work begins. Cost is ~20x higher ($200 vs $9 for a 6-hour session), but the quality delta for product-facing UIs is dramatic. The week's parallel agent work on zhengyi-he-onsite also confirmed this indirectly — agents working without independent evaluation produced code that required significant manual conflict resolution and quality verification.

**Applicability:** Any future multi-feature app sprint should evaluate the harness cost/benefit. Solo Claude is appropriate for backend plumbing and API work; harness is appropriate for frontend and product-quality UX.

### 2. Parallel worktree development works well for additive features but requires explicit coordination architecture

The zhengyi-he-onsite session (20-item backlog, 4 batches × 3 agents) validated parallel worktree development as highly productive for additive features — new columns, new UI cards, new endpoints. However, it revealed two critical coordination needs:

- **Magnet files:** Certain files (`extractionSummary.tsx`, `extractTab.tsx`) are touched by nearly every feature. Identifying these upfront and planning sequential merge passes between batches is essential. Merging all worktrees at once compounds conflicts beyond 3-way merge recovery.
- **Additive vs cross-cutting heuristic:** Features touching ≥3 system layers and modifying shared components (like BL-003: amendments with reconciliation API + PackageReconciliation component + Packages tab) must be kept in-session. The delegation decision should be made before dispatching agents, not after conflicts appear.

**Applicability:** When planning batch agent work, audit the backlog for magnet files and cross-cutting items first. Sequence cross-cutting items before or between agent batches.

### 3. Anthropic SDK file uploads require `MultipartField<InputStream>`, not raw `byte[]` — and Flyway needs special handling in worktree-based development

Two technical footguns consumed significant debugging time this week:

- **Files API:** The Anthropic Java/Kotlin SDK (v2.17.0) silently rejects raw `byte[]` uploads with "Invalid file upload." The correct approach: `MultipartField<InputStream>` with explicit `filename` and `contentType`. The SDK's `FileUploadParams` doesn't expose `contentType` metadata at all (confirmed via `javap` decompilation). This error message gives no hint about the multipart vs byte[] distinction.
- **Flyway + worktrees:** When multiple branches add migrations concurrently, version collisions occur (e.g., two branches both claiming V60). The pattern: rename the later migration to skip occupied versions (V60 → V63). Make migrations idempotent (`IF NOT EXISTS`) to survive manual-apply scenarios. When `flyway_schema_history` is absent, start with `--spring.flyway.enabled=false`, apply manually, then insert the history row.

**Applicability:** For any future Anthropic SDK file work, start with `MultipartField<InputStream>`. For worktree branches with migrations, reserve version numbers upfront in main.
