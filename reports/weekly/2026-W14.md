### 2. GAN-inspired multi-agent harness: separate Generator from Evaluator
**Category:** Architecture / Process

The most impactful architectural insight of the week. An independent evaluator agent — running Playwright MCP and grading against a pre-defined sprint contract — produces dramatically better quality than solo Claude. Solo Claude identifies issues and then talks itself into dismissing them. The evaluator agent can't do that; it just runs the app, checks the contract, and scores pass/fail. Sprint contract pattern: generator proposes scope + acceptance criteria → evaluator reviews before build begins → test against contract after build. Cost ~$200 over 6 hours (vs $9 solo), but justified for serious product-quality frontend or multi-feature work.

**Why it matters:** For any non-trivial build session, the harness investment pays back in reduced rework and higher design quality.

---

### 2. Parallel worktree dev needs upfront magnet-file identification + batched merge passes
**Category:** Process

When dispatching many background agents via git worktrees, certain files are "magnets" — nearly every agent touches them (primary layout, root component, main data transformation file). Without planning for this, you face N-way merge conflicts that exceed standard 3-way merge tooling. The working pattern: identify magnet files before dispatch, run agents in batches of 3, do a full conflict-resolution merge pass before launching the next batch. Trying to merge all worktrees at the end compounds conflicts beyond recovery.

**Why it matters:** Parallel worktree development is highly efficient for additive features but degrades catastrophically without merge coordination. Batch size and explicit merge passes are the key controls.

---

### 3. (Carried from prior week) Additive vs cross-cutting is the right delegation heuristic for background agents
**Category:** Process / Collaboration

When deciding whether to delegate a feature to a background agent or implement in-session: additive = safe to delegate (new column, new UI card, new standalone endpoint). Cross-cutting = keep in-session (feature that modifies ≥3 system layers and touches shared components). Cross-cutting features delegated to agents produce conflicting implementations and architectural drift that takes more time to fix than doing it directly. BL-003 (lease package amendments: new DB tables + reconciliation API + PackageReconciliation component + Packages tab) was the concrete example that surfaced this rule.

**Why it matters:** Over-delegating cross-cutting work is the primary failure mode in parallel worktree sprints.
