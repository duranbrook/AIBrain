## Top 3 Weekly Learnings

1. **Subagent-Driven Development has hit unprecedented throughput and the bottleneck is no longer code authoring — it's owner-gated review/approval cycles.** This week shipped, conservatively counting only merged or implementer-done sub-projects: a 9-task sleek redesign + 5-task dashboard (Fri) + 10-task settings panel + 15-task iOS technician chat + 9-task homepage/login/demo (Sat) + 6-task multi-track Thursday voice/reports/PATCH-estimate + 8-task code-review fix pass — call it ~60 tasks across 7 sub-projects in a single week, all via the design-spec → implementation-plan → subagent-tasks → final-review pattern. Two-stage review is now load-bearing: each task's per-subagent review catches per-file issues, but the *pre-merge final review* keeps catching integration bugs the per-task reviews miss (Saturday: `stripe_secret_key` field-name mismatch + `AccountSection` setTimeout leak; Friday: 8 code-quality issues including React updater purity, blob URL leak, URL-safe base64 decoding, `expire()`-before-`refresh()` ordering, missing test fixtures). What's slowing the cycle now isn't writing code — it's owner-gated steps: Google Cloud Console OAuth client ID setup, Railway migration confirmation, demo-data seed trigger, physical-device camera/voice testing, deploy-state validation. **Strategic implication for W19:** optimize the review/approval loop, not the implementation loop. Possible levers: (a) automate Railway deploy confirmation via `railway logs` parsing as part of the execute trigger; (b) batch owner-gated checkpoints into a single end-of-day decision page rather than scattering them through the backlog; (c) push physical-device testing to a dedicated non-development time slot. Sources: 2026-05-01 daily, 2026-05-02 daily late-sync addendum, 2026-04-30 daily.

2. **Production deploy infrastructure has its own state space that diverges silently from local builds — 4 distinct deploy-only failure modes hit prod in 4 days.** All four bugs were invisible until the artifact reached the deploy target. (i) **Vercel project framework misconfig (Thu 04-30):** `framework: null` silently downgrades a Next.js deploy to `@vercel/static-build`, deploying only `public/` SVGs and 404'ing every dynamic route. (ii) **`lightningcss-darwin-arm64` in `package.json` (Thu 04-30):** macOS-only native binary broke the Linux Vercel build. (iii) **`web/vercel.json` (Fri 05-01):** an `ignoreCommand`-only file in `web/` broke Next.js `app/` directory detection because Vercel root was already configured to `web/`. (iv) **asyncpg UUID hex-only requirement (Fri 05-01):** asyncpg's UUID column adapter rejects mnemonic-prefix UUIDs (`v...`, `j...`) and demands hex-only. The unifying lesson: **deploy targets have invisible state** (Vercel project Framework field, native-binary platform suffixes, GitHub-integration linkage state, database driver type-coercion rules) that doesn't live in the source tree. Two takeaways worth keeping: **(a)** for any Vercel-deployed Next.js app, verify `Framework=nextjs` in the Vercel project settings before debugging routes — `null` is silently destructive for SSR apps and the symptom is "404s in prod, works in `vercel dev`"; **(b)** any package with platform-suffixed binary names (`lightningcss-*`, `esbuild-*`, `@swc/core-*`) is a Linux-vs-macOS deploy hazard and should be filtered from `package.json` if it leaked in via `--save`. Across the week the cluster also produced 4 new memory entries (`feedback_vercel_nextjs_subdir.md`, `feedback_react_infinite_loop_debugging.md`, plus the implicit ones in 04-30's daily), so the "shipped ≠ smoke-tested" budget warning from W17 is now a 4th-instance pattern. Sources: 2026-04-30 daily Learning 1, 2026-05-01 daily Learning 1+supplementary top-3.

The discipline is now baked into the W18 working pattern.
- **Optimistic UI in clients cannot paper over backend exceptions** — API endpoints calling flaky external integrations (Twilio/SendGrid) must isolate them in `try/except` and persist regardless. Tuesday's `6307b71` fix is the canonical example.
- **Concurrent AIBrain sessions race on `.git/*.lock` files held by macOS Xcode source control** — recovery is `mv` (not `rm -f`, fuse mount blocks delete) to `.lock.stale.<ts>`. Pre-flight `lsof` check is the right gate.
- **Final-review pass before merge keeps catching real bugs that per-task reviews miss** — settings panel + 8-issue code-quality fix pass on Thursday both prove this. The pattern is now load-bearing.
