### 2. "Shipped" is not "smoke-tested" — a cross-stack consumer is the cheapest drift detector you have

This week made the case for end-to-end smoke testing twice, in different codebases, with the same lesson:

- **Quote Agent (Apr 24 evening):** 45 unit tests passed; Joe's evening smoke test surfaced four runtime bugs that the suite had missed: argument-order mistake in `create_quote` (`3f143cc`), `QuoteSummary` using the wrong qty field + missing refetch after finalize (`be29322`), agent stream error handler silently swallowing exceptions (`191fbd0`), `quoteId` not restored from chat history on reload (`c11c535`). All four are cross-component data-flow or state-persistence bugs — the kind unit tests structurally don't catch.
- **Mobile Chat (Apr 25):** The pitagents backend had **95 unit tests green** when mobile development started. Yet within the first integration attempt, the iOS and Android clients revealed undocumented backend enum values: direction is `"out"/"in"` (not `"outbound"/"inbound"`), channel is `"wa"/"email"` (not `"sms"/"email"`). No same-language unit test would have caught this — the API contract was self-consistent within the backend's own consumers.

The generalizable rule: **as soon as a backend has its first non-test consumer in a different language or runtime, run a cross-stack smoke call.** It's a far cheaper drift detector than another batch of same-language unit tests, and it surfaces the class of bugs (enum drift, undocumented field shapes, multi-step flow state) that unit tests structurally miss.

**Applicability:** Add cross-stack smoke calls to the definition of done for any backend with multi-client consumption. For pitagents specifically, this means the API contract docs (enum values + endpoint catalog) — already on this week's P2 — are not optional polish but a primary deliverable.

### 3. An 11-day stalled P1 is a signal to demote, not to push harder — and the demote trigger should be momentum elsewhere, not the task itself

car-parts contact enrichment had been P1 since 2026-04-11. By Apr 22 morning, it had carried over for 11 consecutive zero-progress days. The morning execute run (Apr 22) formally demoted it to P2, not because anything had changed about the task — it hadn't — but because a different project (stlab) had just produced real momentum. Backlog hygiene beat backlog sentimentality.

The follow-up move was equally important: when car-parts was revisited on Apr 25, it was **re-scoped before being re-promoted**. Instead of "enrich 3,623 recyclers," the scope became "top-6 chains, 30-minute timebox" — and that scope shipped (176 entries enriched, committed as `093f561`). The remaining 2,000+ per-location entries went back to P2 with a clearer next step ("investigate Google Places API or similar") rather than as a single monolithic task.

**Applicability:** The decision rule worth remembering — **when a task has been P1 for ≥2 weeks with no commits AND there is genuine momentum on a different project, demote on that signal alone. Re-scope before re-promoting.** This is the second time in W17 that a stale-P1 demotion turned into productive work later (the first was on Apr 22; the second was the reduction to top-6 chains). Two data points isn't a law, but it's enough to make this the default playbook.
